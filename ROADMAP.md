# foji-bsd Roadmap

## Shortcuts to revisit

- Local QEMU builds use official FreeBSD `BASIC-CLOUDINIT-zfs` images and persistent qcow2 overlays under `FOJI_FORGE_DIR`. Default image downloads are now pinned by SHA512, but the dedicated build machine should still document the checksum refresh process for future FreeBSD releases.
- The local builder host currently depends on distro package names such as `qemu-full`, `edk2-ovmf`, `edk2-aarch64`, `cdrtools`, `openssh`, `openbsd-netcat`, `rsync`, `hut`, `curl`, and `xz`. A host readiness check now validates commands, firmware paths, KVM access, and active VirtualBox conflicts; convert that checklist into real IaC once the dedicated build machine target distribution is fixed.
- The dedicated builder should avoid running VirtualBox guests while QEMU/KVM package builds are active. On altair, active VirtualBox guests caused `KVM_CREATE_VM` to fail until they were shut down.
- `PKG_REPO_SIGNING_KEY_B64` is currently required for local non-publishing builds because repository signing happens before the optional GitHub release upload. Decide whether this remains desirable or whether unsigned local smoke builds should be supported explicitly.
- The local poudriere build fetches dependencies from the upstream FreeBSD binary package repository when poudriere considers that acceptable. Both local builder architectures currently use `quarterly` plus temporary 2026Q2 `PORTS_REF` pins so `lang/rust` and its dependency closure are fetched instead of built. Revisit these pins as the upstream package sets move.
- `scripts/show-ports-ref-pins.sh` reports the current temporary `PORTS_REF` pins beside the live FreeBSD quarterly branch head. Use it before changing pins, then prove the new pin still allows binary dependency fetches.
- Poudriere may still rebuild `ports-mgmt/pkg` when the fetched binary does not satisfy package-fetch acceptance rules. This is acceptable while `pkg` build time is small; use `POUDRIERE_BULK_FLAGS="-vv"` for future diagnosis before changing blacklist or whitelist policy.
- SourceHut Pages is now the primary package repository publication target because it preserves the flat pkg repository layout. GitHub release publication remains a compatibility path and still requires filtered custom runtime packages because GitHub release assets normalize comma-containing filenames that use epochs such as `brotli-1.2.0,1.pkg`.
- Rust-based `amd64` ports now use a rustup-managed build toolchain instead of depending directly on `lang/rust`; `aarch64` cannot use FreeBSD `devel/rustup-init` today because that port is `amd64`-only. Revisit aarch64 Rust build strategy if upstream rustup host support changes.
- `databases/manticore` is intentionally `amd64`-only for now because it is intended for server nodes and made the first aarch64 builder run impractically long. Revisit if an aarch64 consumer appears.
- The local builder supports targeted port input through `REQUESTED_PORTS`, `FOJI_BUILD_PROFILE=kunci` sets the known-safe lightweight publication path, and `FOJI_BUILD_PROFILE=changed` selects custom ports changed since `CHANGED_SINCE` for local iteration.
- GitHub Actions no longer performs FreeBSD package builds. Keep GitHub as a read-only daily mirror only; SourceHut is the primary source control and release target.
- The repository public key is distributed out of band by sysbsd, not from the SourceHut Pages package repository. Keep this trust flow explicit when adding node bootstrap automation.
